BEM: Organización de CSS

Sabés escribir CSS, pero ¿sabés organizarlo? A medida que un proyecto crece, los estilos se vuelven un caos si no hay una convención de nomenclatura clara. BEM (Block, Element, Modifier) es la metodología más usada para mantener tus clases CSS predecibles, planas y reutilizables. En esta sección vamos a ver por qué usar class en vez de id para estilos, cómo funciona BEM pieza por pieza, y qué errores comunes evitar. Es la diferencia entre un CSS que se mantiene solo y uno que te da pesadillas cada vez que tocás algo.

Class vs ID: la regla de oro

Antes de entrar a BEM, necesitamos dejar claro un punto fundamental que muchos pasan por alto: usá class para estilos y id para funcionalidad (JavaScript). Esta no es solo una preferencia estética, es una regla práctica con consecuencias reales en la mantenibilidad de tu proyecto.

¿Por qué class para estilos?

Un id tiene que ser único en todo el documento HTML. Si ponés id="card" en un componente, no podés tener otro card en la misma página. Esto está bien para un header único, pero ¿qué pasa cuando tenés 10 tarjetas, 3 modales y 5 botones? Los id te obligan a inventar nombres únicos como card-1, card-main, card-sidebar, y el CSS se convierte en un mapa desordenado de excepciones. Las class, en cambio, son reutilizables por definición: podés aplicar la misma clase a cuantos elementos necesites.

Además, los selectores por id tienen más especificidad (0,1,0,0) que los de clase (0,0,1,0). Esto significa que si más adelante querés sobreescribir un estilo definido con #mi-componente, vas a necesitar otro id o, peor aún, !important. Con clases, la especificidad se mantiene plana y cualquier override es simple y predecible.

HTML 
<!-- MAL: id para estilos -->
<div id="card" class="container">
  <h3 id="card-title">Título</h3>
  <p id="card-text">Contenido...</p>
</div>

<!-- BIEN: class para estilos, id solo si JS lo necesita -->
<div class="card" id="js-user-card">
  <h3 class="card__title">Título</h3>
  <p class="card__text">Contenido...</p>
</div>

¿Cuándo usar id entonces?

Los id son perfectos para cuando JavaScript necesita un único punto de referencia: un modal que se abre con un botón, un formulario que se valida, o un elemento que recibe scroll programático. La clave es que el id existe para la lógica, no para el estilo. Si un elemento necesita un id para JS, perfecto — pero el CSS debería seguir apuntando a clases. Un patrón común es usar id con prefijo js- para que sea explícito que ese id es para JavaScript y no para CSS.

HTML 
<!-- El id es para JS, las class son para CSS -->
<button class="btn btn--primary" id="js-open-modal">
  Abrir modal
</button>

<div class="modal" id="js-modal">
  <div class="modal__overlay"></div>
  <div class="modal__content">
    <h2 class="modal__title">Confirmar acción</h2>
    <button class="btn btn--danger" id="js-confirm-delete">Eliminar</button>
  </div>
</div>

Regla práctica

CSS → siempre class. JS → id o data-* attributes. Nunca uses un selector de id (#algo) en tu CSS. Si te encontrás escribiendo #header { ... }, pará y cambiálo a .header { ... }. Tu yo del futuro te lo va a agradecer.

¿Qué es BEM?

BEM es una convención de nomenclatura para clases CSS creada por el equipo de Yandex (el "Google ruso"). El nombre viene de tres conceptos: Block (bloque), Element (elemento) y Modifier (modificador). La idea es simple: cada clase te dice exactamente qué es, dónde vive y qué variante tiene, solo con leerla. No hay que adivinar, no hay que buscar en el HTML, no hay que cruzar los dedos.

BEM resuelve un problema muy concreto: en un proyecto grande, sin convención alguna, terminás con clases como .title, .text, .container, .wrapper que se usan en 15 lugares distintos con estilos completamente diferentes. Cuando querés cambiar el título del card, tocás .title y se rompe el título del modal, el del sidebar y el del footer. BEM elimina esa ambigüedad: .card__title solo afecta al título del card, y punto.

La convención usa dos separadores:

Separador Significado Ejemplo
__ (doble guion bajo) Conecta un Block con su Element card__title
-- (doble guion medio) Agrega un Modifier a un Block o Element card--dark, card__title--large

BEM no es una librería

BEM no es algo que instales con npm install. Es una convención, un acuerdo de equipo sobre cómo nombrar las cosas. No hay código mágico detrás. El valor está en la consistencia: si todo el equipo usa las mismas reglas, cualquier persona puede leer una clase y saber inmediatamente qué componente toca y qué variante es.

Block (Bloque)

Un Block es un componente funcional independiente. Piensalo como un "ladrillo" de tu interfaz: un botón, una tarjeta, un menú, un header, un formulario de búsqueda. La característica clave es que un bloque no depende de su contexto: deberías poder mover el bloque a cualquier parte de la página y seguiría funcionando y viéndose igual.

El nombre del bloque es siempre una palabra o frase corta, sin abreviaciones oscuras. Algunos ejemplos: .card, .menu, .button, .search-form, .user-profile, .price-tag. Se escribe en minúsculas y las palabras se separan con un guión simple (-).

CSS 
.card {
    background: var(--surface-color);
    border: 1px solid var(--border-color);
    border-radius: 8px;
    padding: 1.5rem;
}

.button {
    display: inline-flex;
    align-items: center;
    gap: 0.5rem;
    padding: 0.75rem 1.5rem;
    border: none;
    border-radius: 6px;
    font-weight: 600;
    cursor: pointer;
    transition: background 0.2s;
}
HTML 
<!-- Cada uno es un Block independiente -->
<div class="card">...</div>
<button class="button">Enviar</button>
<form class="search-form">...</form>
<nav class="menu">...</nav>

Nota algo importante: en el HTML, el bloque es el elemento más externo del componente. No hay un wrapper genérico alrededor. El bloque ES el contenedor. Esto es intencional: evita crear divs innecesarios (el famoso "divitis") y mantiene el DOM limpio.

Element (Elemento)

Un Element es una parte interna del bloque. No tiene sentido fuera de él: un .card__title no existe sin un .card, un .menu__item no existe sin un .menu. La relación es de pertenencia: el elemento vive dentro del bloque y solo dentro del bloque. El nombre se escribe después del bloque, separado por doble guion bajo (__).

CSS 
.card__title { font-size: 1.25rem; font-weight: 700; margin-bottom: 0.75rem; }
.card__text { color: var(--text-secondary); line-height: 1.6; margin-bottom: 1rem; }
.card__footer { display: flex; justify-content: space-between; padding-top: 1rem; border-top: 1px solid var(--border-color); }

.menu__item { display: block; padding: 0.75rem 1rem; color: var(--text-secondary); text-decoration: none; border-radius: 4px; transition: background 0.2s; }
.menu__item:hover { background: var(--surface-hover); color: var(--text-primary); }

Nunca anides elementos en BEM

BEM es plano por diseño. Aunque en HTML el elemento esté anidado dentro del bloque, en CSS el selector siempre es una única clase. Nunca escribas .card .card__title ni .card__footer .card__button. El nombre completo ya incluye la relación con el bloque.

Estructura visual: Block + Element
.card
.card__title
.card__text
.card__footer

Modifier (Modificador)

Un Modifier es una variante o estado de un bloque o elemento. No es un componente nuevo, es el mismo componente con un cambio visual o de comportamiento: un botón que es primario en vez de secundario, un card que es oscuro en vez de claro. El modificador se agrega como una clase adicional, nunca reemplaza la clase base. El separador es el doble guion medio (--).

CSS 
.button--primary { background: var(--color-primary); color: white; }
.button--secondary { background: transparent; color: var(--color-primary); border: 1px solid var(--color-primary); }
.button--danger { background: var(--color-danger); color: white; }
.card__title--large { font-size: 1.75rem; }
.menu__item--active { background: var(--color-primary); color: white; }
HTML 
<!-- El bloque base SIEMPRE se mantiene junto al modificador -->
<button class="button button--primary">Guardar</button>
<button class="button button--secondary">Cancelar</button>
<button class="button button--danger">Eliminar</button>

<!-- Modificador en elemento -->
<div class="card">
  <h3 class="card__title card__title--large">Título grande</h3>
</div>

Nunca uses el modificador solo

Si escribís <button class="button--primary"> sin button, el botón no tiene padding, ni border-radius, ni font-weight. El modificador solo cambia propiedades, no reemplaza al bloque. Siempre: class="bloque bloque--modificador".

Estructura visual: Block + Element + Modifier
.button .button--primary .button--danger
.card .card--dark
.card__title .card__title--large
.card__text

Para que se vea cómo funciona todo junto, acá va un componente .user-card con sus elementos y modificadores. Cada clase es autodocumentada: no necesitás leer el HTML para saber qué hace cada una, el nombre lo dice todo.

Antes vs Después: el problema real

Para entender por qué BEM vale la pena, comparemos el mismo componente escrito de las dos formas. Este es el tipo de CSS que todos hemos escrito alguna vez sin una convención, y que al cabo de un par de meses se vuelve inmanejable.

El “antes”: selectores anidados caóticos

CSS 
/* Quién sabe qué '.title' está tocando? */
.card .title { font-size: 1.2rem; }
.sidebar .title { font-size: 0.9rem; }
.modal .title { font-size: 1.5rem; }

/* Especificidad escalando sin control */
.card .container .info p { color: #666; }
.card .container .info p span { font-weight: bold; }
.card.dark .container .info p span { color: white; }
.sidebar .title { color: blue !important; }

El “después”: BEM plano y predecible

CSS 
.card__title { font-size: 1.2rem; }
.sidebar__title { font-size: 0.9rem; }
.modal__title { font-size: 1.5rem; }

/* Especificidad siempre (0,0,1,0) */
.card__text { color: var(--text-secondary); }
.card__text-highlight { font-weight: bold; }
.card--dark .card__text { color: white; }
.card--dark .card__text-highlight { color: rgba(255,255,255,0.9); }
Comparación de especificidad
Sin BEM
.card .title0,0,1,1
.card .container .info p0,0,2,2
.card.dark .container .info p span0,0,3,3
Con BEM
.card__title0,0,1,0
.card__text0,0,1,0
.card--dark .card__text0,0,2,0

Con BEM, todos los selectores base tienen la misma especificidad: (0,0,1,0). El orden en que escribís las clases importa de forma predecible. No hay sorpresas. Si necesitás un override, el modificador con dos clases siempre gana contra uno solo. Simple, consistente, sin !important.

Reglas y errores comunes

Estas son las reglas fundamentales de BEM y los errores más frecuentes al empezar.

Regla¿Bien?¿Mal?
Selectores siempre planos.card__title.card .card__title
El modificador acompaña al baseclass="btn btn--primary"class="btn--primary"
Nombre del bloque: minúsculas + guión.search-form.searchForm
Elemento siempre pertenece a un bloque.card__footer.__footer
No elementos de elementos.card__close-btn.card__footer__btn
Usar class, nunca id para estilos.header#header
Separadores correctos.menu__item--active.menu__item-active

El error más común: elementos anidados

Cuando la gente empieza con BEM, el error número uno es crear elementos de elementos. Si tenés un .card con un .card__footer que a su vez tiene un botón, la tentación es escribir .card__footer__button. No lo hagas. BEM solo permite un nivel de anidamiento (bloque → elemento). Si necesitás un botón dentro del footer, usá un nuevo bloque: .btn. Los bloques se pueden anidar libremente dentro de otros bloques sin romper la convención.

¿Cuándo BEM no es la respuesta?

BEM brilla en CSS vanilla sin herramientas. Pero el ecosistema moderno ofrece alternativas que resuelven el mismo problema de otra forma:

CSS Modules (React/Next.js): genera clases únicas automáticamente (como _card_2x3f1), así que no necesitás una convención de nombres. El scope es automático por archivo. CSS-in-JS (styled-components, Emotion): los estilos viven en JavaScript como strings, se inyectan en runtime y también tienen scope automático. Utility-first (Tailwind CSS): en vez de nombrar componentes, aplicás clases atómicas directamente (flex items-center gap-4 p-5 rounded-xl). No hay que inventar nombres, pero el HTML se vuelve más verboso.

BEM sigue siendo útil

Incluso si usás Tailwind, entender BEM te hace mejor desarrollador porque te enseña a pensar en componentes y especificidad plana. Los conceptos de BEM (bloques independientes, selectores planos, modificadores como variantes) se aplican en cualquier paradigma de CSS.
